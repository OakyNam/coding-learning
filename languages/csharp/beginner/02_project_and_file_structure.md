# 02 - Project and File Structure

## Learning Goal

Learn to recognize the important files in a C# console project, understand what each file or folder is for, and use `dotnet new`, `dotnet build`, and `dotnet run` with confidence.

By the end, you should be able to answer:

- Which file contains the program's starting code?
- Which file tells .NET what kind of project this is?
- Which folders are generated by the build tools?
- Where should you add your own C# code?

## Why Project Structure Matters

A C# console app is usually a project, not just one loose file. The project file tells .NET what to compile, which target framework to use, and how the app should be built and run.

Knowing the structure helps you avoid editing generated files by accident. Most beginner work happens in `.cs` source files such as `Program.cs` and helper files you create. The `bin/` and `obj/` folders are build output folders created by the .NET tools; you usually inspect them only when troubleshooting.

## Create a Console Project

From a terminal, run:

```shell
dotnet new console -n ProjectTour
cd ProjectTour
```

`dotnet new console` creates a new console application. The `-n ProjectTour` option names both the folder and the project.

After creation, the project usually looks like this:

```text
ProjectTour/
|-- ProjectTour.csproj
|-- Program.cs
|-- bin/
`-- obj/
```

You might not see `bin/` until after the first build or run. You might see `obj/` immediately because project restore can create intermediate files.

## Important Files and Folders

`ProjectTour.csproj` is the project file. It tells .NET that this is a C# project, what kind of output to produce, and which target framework to compile against.

`Program.cs` is the default source file. In a new console app, this file usually contains the code that runs first.

`bin/` contains build output, such as compiled assemblies. You normally do not edit files here.

`obj/` contains intermediate build files used by the SDK. You normally do not edit files here either.

## Program.cs and Top-Level Statements

Modern C# console templates use top-level statements. That means a beginner program can start with a statement directly in `Program.cs`:

```csharp
Console.WriteLine("Hello, World!");
```

You do not need to write a `Program` class or a `Main` method for this style. The compiler creates the entry point behind the scenes.

Only one file in a project can contain top-level statements. Other `.cs` files can contain classes, records, structs, interfaces, enums, and other type definitions, but they should not also contain loose executable statements at the top level.

You may still see older examples that write the entry point explicitly:

```csharp
using System;

class Program
{
    static void Main()
    {
        Console.WriteLine("Hello, World!");
    }
}
```

Both styles are valid. For new beginner console apps, top-level statements are shorter and are the default in current templates.

## The Project File

Open `ProjectTour.csproj`. A simple SDK-style console project looks similar to this:

```xml
<Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <TargetFramework>net8.0</TargetFramework>
    <ImplicitUsings>enable</ImplicitUsings>
    <Nullable>enable</Nullable>
  </PropertyGroup>

</Project>
```

Your `TargetFramework` may differ. For example, you might see `net8.0`, `net9.0`, `net10.0`, or another target depending on the SDK and template version installed on your machine.

Important properties:

- `Sdk="Microsoft.NET.Sdk"` tells MSBuild to use the standard .NET SDK project rules.
- `OutputType` set to `Exe` means the project builds an executable app.
- `TargetFramework` tells .NET which framework APIs your project targets.
- `ImplicitUsings` can automatically include common `using` directives for you.
- `Nullable` enables nullable reference type warnings, which help catch possible `null` mistakes.

## Adding Source Files

SDK-style projects include `.cs` files in the project folder by default. You usually do not need to manually list each source file in the `.csproj`.

For example, create a new file named `Greeting.cs`:

```csharp
public static class Greeting
{
    public static string For(string name)
    {
        return $"Hello, {name}!";
    }
}
```

Then update `Program.cs`:

```csharp
Console.WriteLine(Greeting.For("Ada"));
```

When you build or run the project, .NET compiles both `Program.cs` and `Greeting.cs` because both are C# source files in the project folder.

## Build vs Run

Use `dotnet build` to compile the project:

```shell
dotnet build
```

This checks your code and writes compiled output under `bin/`. It also uses intermediate files under `obj/`.

Use `dotnet run` to build if needed and then run the app:

```shell
dotnet run
```

During early learning, `dotnet run` is convenient because it shows whether the program actually behaves the way you expected. `dotnet build` is useful when you want to check for compiler errors without running the app.

## Common Mistakes

- Running `dotnet run` from the wrong folder. Run it from the folder that contains the `.csproj` file, or use `dotnet run --project path/to/project`.
- Editing files inside `bin/` or `obj/`. These folders are generated and can be recreated by the build tools.
- Adding top-level statements to more than one `.cs` file. Keep top-level executable statements in one file, usually `Program.cs`.
- Assuming every new `.cs` file must be listed in the `.csproj`. SDK-style projects include normal source files automatically.
- Changing `TargetFramework` to a version your installed SDK does not support. If that happens, install the matching SDK or choose a target your SDK supports.
- Forgetting that file names and class names are separate ideas. A file named `Greeting.cs` can contain `public static class Greeting`, but the compiler cares about the declared class, not only the file name.

## Exercise

Create and inspect a small console project.

1. Create a project with `dotnet new console -n ProjectTour`.
2. Move into the project folder with `cd ProjectTour`.
3. Open `ProjectTour.csproj` and identify `OutputType`, `TargetFramework`, `ImplicitUsings`, and `Nullable`.
4. Replace `Program.cs` so it calls `Greeting.For("Ada")`.
5. Create `Greeting.cs` with a `Greeting` helper class.
6. Run `dotnet build`.
7. Run `dotnet run`.
8. Write one sentence explaining why `Greeting.cs` did not need to be manually listed in `ProjectTour.csproj`.

## Worked Answer

Commands:

```shell
dotnet new console -n ProjectTour
cd ProjectTour
```

Example `ProjectTour.csproj`:

```xml
<Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <TargetFramework>net8.0</TargetFramework>
    <ImplicitUsings>enable</ImplicitUsings>
    <Nullable>enable</Nullable>
  </PropertyGroup>

</Project>
```

Your `TargetFramework` value may be different. That is okay if your installed SDK supports it.

Replace `Program.cs`:

```csharp
Console.WriteLine(Greeting.For("Ada"));
```

Create `Greeting.cs`:

```csharp
public static class Greeting
{
    public static string For(string name)
    {
        return $"Hello, {name}!";
    }
}
```

Build and run:

```shell
dotnet build
dotnet run
```

Expected output from `dotnet run`:

```text
Hello, Ada!
```

`Greeting.cs` did not need to be manually listed in the project file because SDK-style .NET projects include `.cs` files in the project folder by default. The project file provides the project settings; the source files provide the code.
